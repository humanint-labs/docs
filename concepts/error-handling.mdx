Every payment provider has its own way of signaling failure. Paystack might return `"insufficient_funds"` while Flutterwave returns `"INSUFFICIENT_BALANCE"` and Kuda returns something different again. If you handled these directly, your error handling and retry logic would need a separate branch for every provider. Capisum eliminates this by mapping all provider-native error signals into a single, stable set of error codes with a consistent envelope structure. You write your error handling once — and it works everywhere.

## The error envelope

When a request fails, Capisum returns an `error` object in the response body. The structure is identical regardless of which provider produced the underlying failure:

```json
{
  "error": {
    "code": "INSUFFICIENT_FUNDS",
    "message": "The customer's account has insufficient funds.",
    "retryable": false,
    "provider_code": "insufficient_funds",
    "provider": "paystack"
  }
}
```

Here are the fields in the error object:

<ResponseField name="code" type="string" required>
  Capisum's normalized error code. This is a stable, uppercase string (e.g., `INSUFFICIENT_FUNDS`, `PROVIDER_TIMEOUT`) that is consistent across all providers. Build your error handling and alerting logic against these codes — they will not change when you swap providers.
</ResponseField>

<ResponseField name="message" type="string" required>
  A human-readable description of the error in plain English. This is suitable for logging and for internal debugging, but should not be displayed verbatim to end users without sanitization.
</ResponseField>

<ResponseField name="retryable" type="boolean" required>
  Indicates whether this error is safe to retry. If `true`, the failure is likely transient and a retry with exponential backoff is appropriate. If `false`, the error is terminal — retrying without fixing the underlying issue will produce the same result.
</ResponseField>

<ResponseField name="provider_code" type="string">
  The raw error code or string returned by the underlying provider before normalization. Useful for debugging edge cases, correlating logs with provider dashboards, or raising support tickets with the provider.
</ResponseField>

<ResponseField name="provider" type="string">
  The slug of the provider that produced this error (e.g., `paystack`, `flutterwave`, `kuda`). Present whenever the error originated from a provider call, rather than from Capisum's own validation layer.
</ResponseField>

## Standard error codes

| Code | Retryable | Description |
|------|-----------|-------------|
| `INSUFFICIENT_FUNDS` | No | The customer's account or wallet does not have enough balance to complete the transaction. |
| `PROVIDER_TIMEOUT` | Yes | The provider did not respond within the expected time window. The transaction may or may not have been processed. |
| `INVALID_ACCOUNT` | No | The account number, IBAN, or wallet address provided is invalid or does not exist at the provider. |
| `UNAUTHORIZED` | No | The API key is missing, expired, or does not have permission to perform this operation. |
| `PROVIDER_UNAVAILABLE` | Yes | The provider is temporarily unreachable or experiencing an outage. Retry after a delay or route to an available provider. |
| `INVALID_AMOUNT` | No | The amount is below the provider's minimum, above its maximum, or formatted incorrectly. |
| `DUPLICATE_REFERENCE` | No | A transaction with this `reference` value already exists. References must be unique per transaction. |
| `VALIDATION_ERROR` | No | The request body failed Capisum's validation. Check the `message` field for details on which field is invalid or missing. |

## HTTP status codes

Capisum uses standard HTTP status codes alongside the error envelope. Use the status code for coarse-grained routing in your error handler, and the `code` field for fine-grained logic:

| Status | Meaning |
|--------|---------|
| `200 OK` | The request succeeded. The response body contains the transaction object. |
| `400 Bad Request` | The request body is malformed or missing required fields. Maps to `VALIDATION_ERROR`. |
| `401 Unauthorized` | The API key is missing or invalid. Maps to `UNAUTHORIZED`. |
| `402 Payment Required` | The transaction was declined by the provider (e.g., insufficient funds, card declined). |
| `422 Unprocessable Entity` | The request is structurally valid but cannot be processed — for example, an invalid account number. |
| `429 Too Many Requests` | You have exceeded the rate limit for this endpoint. Back off and retry after the `Retry-After` header value. |
| `500 Internal Server Error` | An unexpected error occurred on Capisum's side or the provider returned an unhandled response. |

## Retryable vs terminal errors

The `retryable` field is the most important signal for your retry logic. Treat errors in one of two ways:

<Tabs>
  <Tab title="Retryable errors">
    If `retryable` is `true`, the failure is transient — a provider timeout, a brief outage, or a network hiccup. It is safe to retry the same request. Use exponential backoff to avoid hammering an already-stressed provider:

    ```typescript
    async function createPayInWithRetry(
      payload: PaymentPayload,
      maxRetries = 3
    ) {
      for (let attempt = 0; attempt < maxRetries; attempt++) {
        try {
          return await capisum.payments.create(payload);
        } catch (err) {
          if (!err.error?.retryable || attempt === maxRetries - 1) {
            throw err; // Terminal error or exhausted retries
          }
          const backoff = Math.pow(2, attempt) * 500; // 500ms, 1s, 2s
          await new Promise((res) => setTimeout(res, backoff));
        }
      }
    }
    ```

    Retryable codes include: `PROVIDER_TIMEOUT`, `PROVIDER_UNAVAILABLE`.
  </Tab>
  <Tab title="Terminal errors">
    If `retryable` is `false`, the error will not resolve by retrying. Something about the request or the underlying data needs to change before you try again. Common actions:

    - **`INSUFFICIENT_FUNDS`** — prompt the customer to top up or use a different payment method.
    - **`INVALID_ACCOUNT`** — ask the customer to re-enter account details.
    - **`DUPLICATE_REFERENCE`** — generate a new unique reference before retrying.
    - **`VALIDATION_ERROR`** — inspect the `message` field and fix the offending request field.
    - **`UNAUTHORIZED`** — rotate or re-supply your API key; do not retry automatically.

    ```typescript
    try {
      const transaction = await capisum.payments.create(payload);
    } catch (err) {
      const { code, retryable, message } = err.error;

      if (!retryable) {
        // Log the failure and surface it to the user or ops team
        logger.error("Terminal payment error", { code, message });
        return { success: false, reason: code };
      }

      // Hand off to retry logic for transient failures
      await scheduleRetry(payload);
    }
    ```
  </Tab>
</Tabs>

## Handling PROVIDER_TIMEOUT carefully

`PROVIDER_TIMEOUT` deserves special attention. When a provider times out, Capisum did not receive a definitive response — the provider may have processed the transaction before the connection dropped. Before retrying a timed-out transaction:

<Warning>
  Do not retry a `PROVIDER_TIMEOUT` with the same reference immediately. First check the transaction status via `GET /v1/transactions/{reference}`. If the transaction is already `processing` or `settled`, the original request succeeded — retrying with the same reference will result in a `DUPLICATE_REFERENCE` error, and retrying with a new reference could create a duplicate charge.
</Warning>

## A note on VALIDATION_ERROR

When Capisum returns `VALIDATION_ERROR` (HTTP `400`), the `message` field will describe the specific field that failed validation, such as:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "The 'currency' field must be a valid ISO 4217 code.",
    "retryable": false,
    "provider_code": null,
    "provider": null
  }
}
```

Note that `provider_code` and `provider` are `null` for validation errors — these failures never reach a provider adapter, so there is no provider-side context to include.

<Tip>
  Log both `code` (Capisum's normalized code) and `provider_code` (the raw provider string) for every error. The `code` is what your alerting rules, dashboards, and retry logic should reference — it's stable across providers and releases. The `provider_code` is indispensable when you're debugging an edge case or filing a support ticket with the underlying provider, because it's exactly what appeared in the provider's own response before normalization.
</Tip>
