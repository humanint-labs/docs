Every transaction in Capisum is associated with exactly one provider, and you decide which one. You specify the provider by name in your request body, and Capisum routes that transaction to the corresponding adapter. There is no ambiguity about which backend ran a call — the provider you set is the provider that executes, and the response always confirms it. This explicit, predictable routing model is deliberate: you stay in control of every payment flow from the start.

## How manual routing works

When you send a request to `POST /v1/transactions`, the `provider` field tells Capisum where to route it. The rest of the request shape — `amount`, `currency`, `type`, `customer`, `reference` — stays identical regardless of which provider you select. You don't need to rewrite your request logic, restructure your payload, or learn a different API dialect per provider. The adapter layer handles the translation behind the scenes.

```json
// Pay via Paystack
{
  "provider": "paystack",
  "amount": 500000,
  "currency": "NGN",
  "type": "pay_in",
  "reference": "your-unique-ref-001",
  "customer": {
    "email": "customer@example.com",
    "name": "Jane Doe"
  }
}
```

```json
// Pay via Flutterwave — same shape, different provider
{
  "provider": "flutterwave",
  "amount": 500000,
  "currency": "NGN",
  "type": "pay_in",
  "reference": "your-unique-ref-002",
  "customer": {
    "email": "customer@example.com",
    "name": "Jane Doe"
  }
}
```

Switching providers in your integration is a one-field change. The response you get back is also identical in shape — Capisum normalizes the provider's native response before returning it to you.

## Checking provider status from the dashboard

Your Capisum dashboard shows the connection status of every provider on your account. Before routing transactions to a provider, you can confirm it's reachable and properly configured.

<Tabs>
  <Tab title="Selected">
    The provider is the active route for this operation type. Transactions with this provider slug in the request body will be dispatched to this adapter.
  </Tab>
  <Tab title="Available">
    The provider is connected and ready to receive transactions. You can switch routing to this provider at any time by updating the `provider` field in your requests.
  </Tab>
  <Tab title="Unavailable">
    The provider is either not yet connected, credentials are missing, or the provider is experiencing an outage. Transactions routed to an unavailable provider will return a `PROVIDER_UNAVAILABLE` error.
  </Tab>
</Tabs>

## Routing the same operation through different providers

A common pattern is to connect multiple providers for the same operation type — for example, both Paystack and Flutterwave for NGN pay-ins — and route different customers or transaction segments to different providers based on your own business logic. Because Capisum's request shape is identical across providers, you can implement this in your application without any provider-specific branching in your integration code:

```typescript
async function createPayIn(amount: number, providerSlug: string) {
  return await capisum.payments.create({
    provider: providerSlug, // "paystack" | "flutterwave" | "monnify" | ...
    amount,
    currency: "NGN",
    type: "pay_in",
    reference: generateReference(),
    customer: {
      email: currentUser.email,
      name: currentUser.name,
    },
  });
}

// Route by segment, merchant tier, or any other logic you control
const transaction = await createPayIn(500000, "flutterwave");
```

Your routing logic lives entirely in your application — Capisum executes whatever provider you specify.

## Smart routing: coming later

In the current release, all routing is explicit and manual. Capisum does not automatically select or switch providers for you. Smart routing — including configurable routing rules, automatic failover, and retry policies — is on the roadmap as an optional dashboard feature. When it ships, you'll be able to define rules like "prefer Paystack for NGN pay-ins, fall back to Flutterwave if unavailable" without changing your application code. Until then, you own the routing logic.

<Accordion title="What happens if I route to a provider that is unavailable?">
  Capisum returns a `PROVIDER_UNAVAILABLE` error immediately, before any charge is attempted. The error envelope includes `"retryable": true`, meaning you can safely retry the request — either after a delay for the same provider or by switching to an available alternative. No funds are moved and no partial state is created on the provider side.
</Accordion>

<Tip>
  Before sending any live transactions to a new provider, test the full routing path in sandbox mode first. Sandbox mirrors the live adapter behavior — including error responses and status transitions — so you can validate your routing logic, reference generation, and error handling end-to-end without touching real money or real provider accounts.
</Tip>
