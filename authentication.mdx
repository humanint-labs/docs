The customer API (`/v1/*`) uses organization API keys. Create keys in the [dashboard](https://app.capisum.com).

## API key format

```
cps_8Kx9mN2pQ7rT4vW1yZ3aB5cD6eF0gH
```

Keys use the `cps_` prefix. The full key is shown once at creation.

## Authorization header

```bash
curl "$CAPISUM_BASE_URL/v1/transactions" \
  -H "Authorization: Bearer cps_your_key_here"
```

| Header | Required | Description |
| --- | --- | --- |
| `Authorization` | Yes | `Bearer cps_...` |
| `Content-Type` | For JSON bodies | `application/json` |
| `X-Idempotency-Key` | For payout creation | See [Idempotency](/guides/idempotency) |

## Staging vs production

Each key is bound to **staging** or **production** at creation:

| Environment | Use case |
| --- | --- |
| Staging | Sandbox provider credentials, test payouts |
| Production | Live provider credentials, real funds |

A staging key never reads or writes production transactions. Create separate keys for each environment.

<Warning>
  Never use production API keys in client-side code, public repositories, or shared logs.
</Warning>

## Errors

| HTTP status | Code | Meaning |
| --- | --- | --- |
| 401 | `UNAUTHORIZED` | Missing or invalid API key |
