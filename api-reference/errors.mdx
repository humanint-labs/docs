When a Capisum API request cannot be completed successfully, the response always contains a structured error envelope in the body. The shape is consistent regardless of which endpoint you called or which provider was involved — your error handling, logging, and alerting logic only needs to be written once. Every error carries a stable Capisum-normalized code alongside the raw provider information, so you can build on `code` values without worrying about provider-specific strings changing underneath you.

## Error Envelope

All errors are returned under a top-level `error` object. The HTTP status code and the `code` field inside the envelope both convey the failure category — use them together for the most accurate handling.

```json
{
  "error": {
    "code": "INSUFFICIENT_FUNDS",
    "message": "The customer's account has insufficient funds.",
    "retryable": false,
    "provider_code": "insufficient_funds",
    "provider": "paystack"
  }
}
```

<Expandable title="Error envelope fields">

<ResponseField name="error.code" type="string" required>
  The Capisum-normalized error code. Stable across all providers and API versions within `v1`. Use this field in your application logic and alerting rules.
</ResponseField>

<ResponseField name="error.message" type="string" required>
  A human-readable explanation of what went wrong. Intended for logs and debugging — do not parse this string programmatically, as wording may change without notice.
</ResponseField>

<ResponseField name="error.retryable" type="boolean" required>
  Indicates whether it is safe to retry the same request. When `true`, the failure was likely transient (e.g., a provider timeout). When `false`, retrying the identical request without changes will not succeed.
</ResponseField>

<ResponseField name="error.provider_code" type="string | null">
  The raw error code returned by the underlying provider before normalization. Useful for cross-referencing provider-specific documentation or support tickets. `null` when the error originated within Capisum before reaching a provider.
</ResponseField>

<ResponseField name="error.provider" type="string | null">
  The provider that returned the error (e.g., `"paystack"`, `"flutterwave"`). `null` when the error originated within Capisum before reaching a provider.
</ResponseField>

</Expandable>

## HTTP Status Codes

Capisum uses standard HTTP status codes. The table below describes when each code is returned.

| HTTP Status | When It Occurs |
|-------------|----------------|
| `200 OK` | Request succeeded |
| `400 Bad Request` | Missing or malformed request body |
| `401 Unauthorized` | Missing or invalid API key |
| `403 Forbidden` | Wrong environment key or insufficient permissions |
| `422 Unprocessable` | Valid JSON but failed business validation |
| `429 Too Many Requests` | Rate limit exceeded |
| `500 Internal Server Error` | Capisum internal error |
| `502 Bad Gateway` | Provider returned an unexpected response |

<Note>
  `400` vs `422`: A `400` means Capisum could not parse your request (e.g., malformed JSON or a missing required field at the schema level). A `422` means your request was structurally valid but failed business rules — for example, submitting a duplicate transaction reference or an amount outside the provider's allowed range.
</Note>

## Error Code Reference

The table below lists every stable error code you may encounter, its associated HTTP status, whether it is retryable, and a short description.

| Code | HTTP | Retryable | Description |
|------|------|-----------|-------------|
| `UNAUTHORIZED` | 401 | No | Invalid or missing API key |
| `FORBIDDEN` | 403 | No | Action not permitted for this key |
| `VALIDATION_ERROR` | 422 | No | Request body failed validation |
| `INSUFFICIENT_FUNDS` | 402 | No | Customer account has insufficient funds |
| `INVALID_ACCOUNT` | 422 | No | Account number or details are invalid |
| `DUPLICATE_REFERENCE` | 422 | No | A transaction with this reference already exists |
| `INVALID_AMOUNT` | 422 | No | Amount is outside provider-allowed limits |
| `PROVIDER_TIMEOUT` | 502 | Yes | Provider did not respond in time |
| `PROVIDER_UNAVAILABLE` | 502 | Yes | Provider is temporarily down |
| `PROVIDER_ERROR` | 502 | No | Provider returned a terminal error |
| `RATE_LIMITED` | 429 | Yes | Too many requests; back off and retry |
| `INTERNAL_ERROR` | 500 | Yes | Capisum internal error; contact support |

<Warning>
  `DUPLICATE_REFERENCE` is always terminal — retrying with the same `reference` value will never create a second transaction. If you need to retry a failed operation, generate a new unique reference for the attempt.
</Warning>

## Retry Strategy

For errors where `retryable` is `true` (such as `PROVIDER_TIMEOUT`, `PROVIDER_UNAVAILABLE`, `RATE_LIMITED`, and `INTERNAL_ERROR`), use **exponential backoff with jitter**. This prevents thundering-herd problems when a provider recovers or when multiple clients hit a rate limit simultaneously.

The recommended defaults are:

- **Base delay**: 1 second
- **Backoff factor**: 2× per attempt (1s → 2s → 4s)
- **Jitter**: add up to 500ms of random delay on each attempt
- **Maximum retries**: 3 before propagating the error to your caller

<Tip>
  Always check `error.retryable` before retrying. Retrying a terminal error (e.g., `INSUFFICIENT_FUNDS`) wastes resources and will never produce a different outcome.
</Tip>

```typescript
async function createWithRetry(params, maxRetries = 3) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      return await capisum.payments.create(params);
    } catch (error) {
      if (!error.retryable || attempt === maxRetries - 1) throw error;
      const delay = Math.pow(2, attempt) * 1000 + Math.random() * 500;
      await new Promise((resolve) => setTimeout(resolve, delay));
    }
  }
}
```

After exhausting all retries, surface the final error to your application layer. For `INTERNAL_ERROR` responses that persist beyond a few minutes, contact Capisum support and include the full error envelope — the `provider_code` and `provider` fields help the support team trace the failure end to end.
