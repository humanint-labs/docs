The Capisum REST API gives you a single, stable integration surface for payment gateways, digital banks, wallets, crypto platforms, and account infrastructure providers. You send one standard request shape, Capisum routes it to the provider you specify, and you receive one normalized response — regardless of what the underlying provider actually returned. Every resource lives under the versioned base URL below, all transport is HTTPS, and all request and response bodies are JSON.

## Base URL

All API requests target the following base URL:

```
https://api.capisum.com/v1
```

<Note>
  There is no trailing slash. Including one may cause requests to be rejected or redirected depending on your HTTP client.
</Note>

## Making Requests

Every request you send must meet these baseline requirements:

- **Transport** — HTTPS only. Plain HTTP connections are refused.
- **Authentication** — Include your API key in the `Authorization` header as a Bearer token. See [Authentication](/api-reference/authentication) for full details.
- **Content-Type** — Set `Content-Type: application/json` on all requests that include a body (`POST`, `PATCH`). Requests without a body (such as `GET`) do not require this header.
- **Responses** — All responses are JSON. Your client should always parse the response body and check the HTTP status code together.

## Monetary Amounts

All monetary amounts throughout the API are expressed as integers in the **smallest unit of the relevant currency**. For example:

- Nigerian Naira (NGN): amounts are in **kobo** (1 NGN = 100 kobo, so ₦5,000.00 = `500000`)
- US Dollar (USD): amounts are in **cents** (1 USD = 100 cents, so $50.00 = `5000`)

Never send decimal values for amounts. The API will reject them with a `VALIDATION_ERROR`.

<Warning>
  Sending amounts in major currency units (e.g., `5000` to mean ₦5,000.00 when the API expects `500000` kobo) is one of the most common integration mistakes. Double-check your unit conversion before going to production.
</Warning>

## Endpoint Index

The table below lists every available endpoint. All paths are relative to the base URL.

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/v1/transactions` | Create a pay-in or payout transaction |
| `GET` | `/v1/transactions/:id` | Retrieve a transaction by ID |
| `GET` | `/v1/transactions` | List transactions with filters |
| `GET` | `/v1/providers` | List available providers |
| `GET` | `/v1/providers/:id` | Get a specific provider by ID |

## Rate Limiting

Rate limits are enforced **per API key**. When you exceed the limit for your plan, the API returns a `429 Too Many Requests` response with a `RATE_LIMITED` error code in the body. The response includes a `Retry-After` header indicating how many seconds to wait before retrying.

<Note>
  Rate limit thresholds vary by plan. Refer to your dashboard or contact support if you need higher limits for your integration volume.
</Note>

For guidance on handling `429` responses gracefully, including exponential backoff logic, see [Errors](/api-reference/errors).

## Versioning

The current stable API version is **`v1`**, reflected in the base URL path. Capisum maintains backward compatibility within a version — additive changes such as new fields and new endpoints are made without notice. Breaking changes (removed fields, changed semantics, new required parameters) will always be announced in advance with a migration guide and a deprecation window before the old version is retired.

<Tip>
  Pin your HTTP client to the `/v1` path explicitly. Do not rely on an unversioned alias resolving to the current version, as that behavior is not guaranteed.
</Tip>

## Quick Example

The following shows a complete pay-in transaction request and its normalized response. You can adapt this to your language of choice using the `Authorization` header and a JSON body.

<CodeGroup>

```bash cURL
curl -X POST https://api.capisum.com/v1/transactions \
  -H "Authorization: Bearer cap_sk_live_..." \
  -H "Content-Type: application/json" \
  -d '{
    "provider": "paystack",
    "type": "pay_in",
    "amount": 500000,
    "currency": "NGN",
    "customer": {
      "email": "customer@example.com"
    },
    "reference": "order-001"
  }'
```

</CodeGroup>

**Response**

```json
{
  "id": "txn_8f2k9m",
  "status": "successful",
  "provider": "paystack",
  "amount": 500000,
  "currency": "NGN"
}
```

Notice that the response fields are identical regardless of which provider you route to. Capisum normalizes the provider's native response into this standard shape before returning it to you.
