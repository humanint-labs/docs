[`POST /v1/transactions/payout`](/api-reference) creates a payout routed through your connected providers. Payouts are the supported transaction type on the API today. This guide covers request fields, the transaction lifecycle, and the steps to go live.

## Go-live checklist

Complete these steps before processing real payouts in production:

<Steps>
  <Step title="Test in sandbox">
    Connect providers with sandbox credentials and run the [Quickstart](/quickstart). Confirm payouts reach `completed` and [webhooks](/guides/webhooks) verify correctly.
  </Step>
  <Step title="Connect live providers">
    Switch the dashboard to **Live** and connect each provider with production API credentials. For Paystack, [disable Transfer OTP](/guides/providers#disable-transfer-otp) in the Paystack Dashboard (required for payouts; account-wide, including test).
  </Step>
  <Step title="Create a production API key">
    Generate a production key in [API keys](https://app.capisum.com/api-keys). Store it server-side only.
  </Step>
  <Step title="Register your webhook URL">
    Add a production endpoint at [Webhooks → Configuration](https://app.capisum.com/webhooks/config). Save the signing secret securely.
  </Step>
  <Step title="Verify beneficiary accounts">
    Call [`POST /v1/verify/account`](/guides/account-verification) before your first live payout to confirm account names.
  </Step>
  <Step title="Start with a small payout">
    Send a low-value transfer to validate end-to-end routing, then scale volume gradually.
  </Step>
</Steps>

See [Providers](/guides/providers) for per-provider capabilities and currency support.

## Amounts

All amounts are in **minor units** (the smallest currency unit):

| Currency | Minor unit | Example |
| --- | --- | --- |
| NGN | Kobo | `500000` = ₦5,000.00 |
| GHS | Pesewa | `10000` = GH₵100.00 |
| KES | Cent | `50000` = KSh500.00 |

Pass `amount` as a string or integer. Capisum stores and returns amounts as integers.

## Supported currencies and rails

| Currency | `bank_account` | `mobile_money` |
| --- | --- | --- |
| NGN | Yes (`bank_id` required) | No |
| GHS | Yes (`bank_id` required) | Yes (`mobile_money_code` required) |
| KES | Yes (`bank_id` required) | Yes (`mobile_money_code` required) |

Use [`GET /v1/reference/banks`](/api-reference) to list canonical `bank_id` values. Capisum resolves provider-specific institution codes internally at payout time.

### Destination object

Use `destination` on create requests. `recipient` is a deprecated alias.

| Field | Required | Description |
| --- | --- | --- |
| `type` | Recommended | `bank_account` or `mobile_money` |
| `account_number` | Yes | Bank account or mobile wallet number |
| `bank_id` | For bank payouts | Capisum canonical bank id (e.g. `ng.opay` for OPay) |
| `mobile_money_code` | For mobile money | GHS: `MTN`, `AIRTEL`. KES: `SAFARICOM`, `AIRTEL` |
| `account_name` | Recommended | Display name for the recipient |
| `country` | No | ISO 3166-1 alpha-2 code. Defaults from currency |
| `phone` | No | Wallet phone. Defaults to `account_number` |

<Warning>
  `bank_code` is a deprecated escape hatch and is not part of the supported contract. Use `bank_id` instead. `bank_code` cannot be combined with `bank_id`.
</Warning>

## References

| Field | Who sets it | Format |
| --- | --- | --- |
| `reference` (request body) | You (optional) | 16–50 chars; lowercase letters, digits, `_`, `-` |
| `reference` (response) | Capisum | Your merchant-facing payout id (`pout_*` when auto-generated) |
| `provider_reference` | Provider | Provider transfer id after submission |

Capisum generates a provider-safe `pout_*` reference when you omit `reference`. On the first provider attempt, that merchant reference is used as the provider transfer idempotency key. Retries after a confirmed failure or reversal use fresh internal transfer references — you only track the merchant `reference` in your integration.

## Provider routing

Set `provider` to route explicitly:

```json
{
  "provider": "paystack"
}
```

When omitted, Capisum selects a provider using your organization's routing rules configured in the dashboard.

<Info>
  Routing rule management is a dashboard operation. The public API only accepts an optional `provider` override on each request.
</Info>

## Transaction lifecycle

Payout creation returns immediately with `status: queued`. Capisum submits the payout to the provider asynchronously in the background.

| Status | Meaning |
| --- | --- |
| `queued` | Accepted; waiting for worker submission |
| `processing` | Provider submission in progress |
| `pending_provider_confirmation` | Submitted to provider; awaiting webhook or confirmation poll |
| `uncertain` | Provider outcome unclear (timeout or ambiguous response) |
| `manual_review` | Needs attention; eligible for dashboard retry after failed attempt |
| `completed` / `settled` | Payout succeeded |
| `failed` | Terminal failure |
| `refunded` | Provider reversed the transfer |
| `cancelled` | Cancelled before settlement |

Prefer [webhooks](/guides/webhooks) for terminal status updates. Capisum also polls the provider after a **1 minute** webhook grace period (plus early confirm polls at ~15s / 30s / 60s). Use verify as a fallback:

- `GET /v1/transactions/{id}` or `GET /v1/transactions/ref/{reference}` — read current status
- `POST /v1/transactions/{id}/verify` — ask the provider for the latest status

<Note>
  Use `status_group=review` when listing transactions to find payouts in `uncertain` or `manual_review`.
</Note>
