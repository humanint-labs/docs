Capisum's payout flow lets you disburse funds to bank accounts and wallets through any connected provider using the same unified API surface you use for pay-ins. Whether you're paying vendors, settling freelancers, processing refunds, or distributing marketplace earnings, you send one standardized request and Capisum routes it to the correct provider adapter. The response shape — and the error codes — are consistent regardless of which provider executes the transfer.

## When to use this guide

Use this guide for any outbound money movement your application initiates: vendor disbursements, customer refunds, salary runs, affiliate payouts, wallet withdrawals, or peer-to-peer transfers. Providers with payout support include Paystack, Flutterwave, Remita, Korapay, SeerBit, and digital banks like Kuda and OPay.

## Prerequisites

Before sending a payout, confirm you have:

- A Capisum API key (`cap_sk_live_...` for production, `cap_sk_test_...` for sandbox)
- A provider connected to your Capisum account with **Payout** support enabled
- Sufficient balance in your provider account to cover the disbursement amount plus any provider fees
- The recipient's verified bank account number, bank code, and full name

<Warning>
  Payouts are irreversible once the provider marks them as settled. Always validate the recipient's account number and bank code before submitting the request — a transfer sent to the wrong account cannot be automatically recalled through Capisum. Use your provider's account name lookup or verification endpoint to confirm recipient details before initiating a payout.
</Warning>

## Steps

<Steps>
  ### Verify the recipient details

  Before creating a payout, confirm that the recipient's bank account information is accurate. Most supported providers expose an account resolution endpoint that returns the registered account name for a given account number and bank code. Cross-reference the resolved name against your records before proceeding.

  Key fields to validate:

  - **`account_number`** — must be a valid account number for the target bank
  - **`bank_code`** — the provider-specific bank sort code (e.g., `"058"` for Guaranty Trust Bank on Flutterwave)
  - **`name`** — the registered account holder name, used to detect mismatches before funds leave your account

  If the resolved name does not match your expected recipient, halt the payout and raise a verification flag in your system before proceeding.

  ### Create the payout transaction

  Send a `POST` request to `/v1/transactions` with `type` set to `"payout"`. Include the provider, amount in the smallest currency unit, currency, recipient details, a unique reference, and an optional `narration` that will appear on the recipient's bank statement.

  <CodeGroup>
    ```bash cURL
    curl -X POST https://api.capisum.com/v1/transactions \
      -H "Authorization: Bearer cap_sk_live_..." \
      -H "Content-Type: application/json" \
      -d '{
        "provider": "flutterwave",
        "type": "payout",
        "amount": 250000,
        "currency": "NGN",
        "recipient": {
          "account_number": "0123456789",
          "bank_code": "058",
          "name": "John Doe"
        },
        "reference": "payout-ref-456",
        "narration": "Vendor payment - January 2024"
      }'
    ```

    ```typescript TypeScript
    const payout = await capisum.payments.create({
      provider: "flutterwave",
      type: "payout",
      amount: 250000,
      currency: "NGN",
      recipient: {
        account_number: "0123456789",
        bank_code: "058",
        name: "John Doe",
      },
      reference: "payout-ref-456",
      narration: "Vendor payment - January 2024",
    });
    ```
  </CodeGroup>

  <Tip>
    The `narration` field controls the description that appears on the recipient's bank statement. Keep it clear and meaningful — something like `"Vendor payment - January 2024"` or `"Refund for order #78901"` reduces recipient confusion and decreases the likelihood of a dispute or callback.
  </Tip>

  A successful creation response returns the transaction record with `status: "processing"`:

  ```json Payout created response
  {
    "id": "txn_9k3m1p",
    "status": "processing",
    "provider": "flutterwave",
    "type": "payout",
    "amount": 250000,
    "currency": "NGN",
    "reference": "payout-ref-456",
    "created_at": "2024-01-15T15:30:00Z"
  }
  ```

  The `"processing"` status means Capisum has accepted and forwarded the request to the provider. The transfer has not settled yet — do not mark it complete in your system at this point.

  ### Poll or check the transaction status

  Most providers process payouts asynchronously. After creating the transaction, poll `GET /v1/transactions/:id` periodically until the status reaches a terminal state (`"successful"`, `"settled"`, or `"failed"`).

  <CodeGroup>
    ```bash cURL
    curl -X GET https://api.capisum.com/v1/transactions/txn_9k3m1p \
      -H "Authorization: Bearer cap_sk_live_..."
    ```

    ```typescript TypeScript
    const payout = await capisum.transactions.get("txn_9k3m1p");
    console.log(payout.status); // "processing" | "successful" | "settled" | "failed"
    ```
  </CodeGroup>

  A reasonable polling strategy is to check every 10–30 seconds for up to 5 minutes, then surface an alert to your operations team if no terminal status is received. Most providers settle payouts within 1–3 minutes during banking hours.

  Payout transaction statuses:

  | Status | Meaning |
  |--------|---------|
  | `processing` | Request submitted to provider, awaiting bank confirmation. |
  | `successful` | Provider confirmed the transfer was accepted. |
  | `settled` | Funds confirmed as received by the recipient's bank. |
  | `failed` | Transfer could not be completed. Check `error.code`. |

  ### Handle completion or failure

  **Successful / Settled:**

  ```json Settled payout response
  {
    "id": "txn_9k3m1p",
    "status": "settled",
    "provider": "flutterwave",
    "provider_ref": "FLW-TXN-00129384",
    "type": "payout",
    "amount": 250000,
    "currency": "NGN",
    "reference": "payout-ref-456",
    "narration": "Vendor payment - January 2024",
    "created_at": "2024-01-15T15:30:00Z",
    "settled_at": "2024-01-15T15:31:47Z"
  }
  ```

  When status is `"successful"` or `"settled"`, update your records, notify the recipient if appropriate, and close the disbursement task.

  **Failed:**

  ```json Failed payout response
  {
    "id": "txn_9k3m1p",
    "status": "failed",
    "provider": "flutterwave",
    "type": "payout",
    "amount": 250000,
    "currency": "NGN",
    "reference": "payout-ref-456",
    "error": {
      "code": "INVALID_ACCOUNT",
      "message": "The recipient account number could not be resolved by the provider.",
      "retryable": false
    },
    "created_at": "2024-01-15T15:30:00Z"
  }
  ```

  When status is `"failed"`, inspect `error.code`. Terminal errors like `INVALID_ACCOUNT` require human review of the recipient's details before any retry. Retryable errors like `PROVIDER_TIMEOUT` can be re-submitted with a new reference after a short backoff.

</Steps>

## Payout error codes reference

| Code | Retryable | Description |
|------|-----------|-------------|
| `INVALID_ACCOUNT` | No | Recipient account number or bank code is invalid. |
| `INSUFFICIENT_FUNDS` | No | Your provider account balance is too low. |
| `PROVIDER_TIMEOUT` | Yes | Provider did not respond — safe to retry with a new reference. |
| `PROVIDER_ERROR` | Yes | Transient provider-side failure. |
| `INVALID_CURRENCY` | No | Provider does not support this currency for payouts. |
| `DUPLICATE_REFERENCE` | No | A transaction with this reference already exists. Use a new reference. |
