Capisum's pay-in flow lets you accept customer payments — cards, bank transfers, virtual accounts, and local payment methods — through any supported provider without changing your integration code. You pick the provider per transaction, send one standardized request to Capisum, and receive one normalized response regardless of which gateway processed the charge. This guide walks you through every step, from selecting a provider to confirming a settled transaction.

## When to use this guide

Use this guide whenever your application needs to collect funds from a customer: e-commerce checkouts, subscription billing, invoice payments, wallet top-ups, or any other inbound money movement. Capisum normalizes pay-in support across Paystack, Flutterwave, Monnify, Korapay, SeerBit, and more — so switching providers later requires only changing the `provider` field in your request.

## Prerequisites

Before you begin, make sure you have:

- A Capisum API key (`cap_sk_live_...` for production, `cap_sk_test_...` for sandbox)
- At least one provider connected to your Capisum account with **Pay-in** support enabled (e.g., Paystack, Flutterwave, Monnify, Korapay)
- Your provider account credentials added and verified in the Capisum dashboard under **Providers**

<Note>
  If you haven't tested your integration yet, work through the [sandbox testing guide](/guides/sandbox-testing) first. Sandbox mode uses the same API shapes with test credentials so no real money moves.
</Note>

## Steps

<Steps>
  ### Select your provider

  Capisum gives you explicit control over which provider handles each transaction. Before creating a transaction, confirm that your chosen provider is connected and supports pay-in operations.

  Navigate to **Dashboard → Providers** to see your connected providers and their supported operations. Look for the **Pay-in** badge on the provider tile. Providers with live pay-in support include:

  <CardGroup cols={3}>
    <Card title="Paystack" icon="credit-card">
      Cards, bank transfers, and local payment methods (NGN).
    </Card>
    <Card title="Flutterwave" icon="credit-card">
      Cards, transfers, and mobile money across multiple currencies.
    </Card>
    <Card title="Monnify" icon="building-columns">
      Virtual accounts and bank transfers (NGN).
    </Card>
    <Card title="Korapay" icon="credit-card">
      Cards, bank transfers, and checkout payments.
    </Card>
    <Card title="SeerBit" icon="credit-card">
      Multi-channel payment collections.
    </Card>
    <Card title="Interswitch" icon="credit-card">
      Card and switch-based payment routing (Beta).
    </Card>
  </CardGroup>

  Once you've confirmed your provider is active, proceed to create the transaction.

  ### Create the transaction

  Send a `POST` request to `/v1/transactions` with the `type` set to `"pay_in"`. Include the amount (in the smallest currency unit — kobo for NGN, cents for USD), the currency, customer details, and a unique reference you control.

  <CodeGroup>
    ```bash cURL
    curl -X POST https://api.capisum.com/v1/transactions \
      -H "Authorization: Bearer cap_sk_live_..." \
      -H "Content-Type: application/json" \
      -d '{
        "provider": "paystack",
        "type": "pay_in",
        "amount": 500000,
        "currency": "NGN",
        "customer": {
          "email": "customer@example.com",
          "name": "Jane Doe"
        },
        "reference": "order-78901"
      }'
    ```

    ```typescript TypeScript
    const transaction = await capisum.payments.create({
      provider: "paystack",
      type: "pay_in",
      amount: 500000,
      currency: "NGN",
      customer: {
        email: "customer@example.com",
        name: "Jane Doe",
      },
      reference: "order-78901",
    });
    ```
  </CodeGroup>

  <Tip>
    Always generate a unique `reference` for every transaction — use your internal order ID, invoice number, or a UUID. Capisum uses this field to deduplicate requests, and most providers will reject a duplicate reference outright. Never reuse a reference across separate transactions.
  </Tip>

  ### Redirect the customer (if applicable)

  Some providers — Paystack and Flutterwave in particular — return a hosted checkout URL in the transaction response. When `checkout_url` is present in the response body, redirect the customer's browser to that URL to complete payment on the provider's checkout page.

  ```json Example response with checkout URL
  {
    "id": "txn_8f2k9m",
    "status": "pending",
    "provider": "paystack",
    "type": "pay_in",
    "amount": 500000,
    "currency": "NGN",
    "reference": "order-78901",
    "checkout_url": "https://checkout.paystack.com/8f2k9m",
    "created_at": "2024-01-15T14:02:11Z"
  }
  ```

  Providers that use virtual accounts (such as Monnify) do not return a checkout URL. Instead, the response contains the virtual account details that you display directly to your customer for them to initiate a bank transfer.

  ### Check the transaction status

  After the customer completes (or attempts) payment, retrieve the current transaction status by polling `GET /v1/transactions/:id`. Replace `:id` with the `id` returned when you created the transaction.

  <CodeGroup>
    ```bash cURL
    curl -X GET https://api.capisum.com/v1/transactions/txn_8f2k9m \
      -H "Authorization: Bearer cap_sk_live_..."
    ```

    ```typescript TypeScript
    const transaction = await capisum.transactions.get("txn_8f2k9m");
    ```
  </CodeGroup>

  A transaction moves through the following statuses:

  <Tabs>
    <Tab title="pending">
      The transaction has been created but the customer has not yet completed payment. Safe to display a "waiting for payment" state.
    </Tab>
    <Tab title="processing">
      Payment has been submitted and the provider is verifying it. Avoid taking fulfillment actions until the status advances.
    </Tab>
    <Tab title="successful">
      The provider has confirmed the payment. For most use cases, this is the signal to trigger fulfillment.
    </Tab>
    <Tab title="settled">
      Funds have been settled to your provider account. This is the terminal success state.
    </Tab>
    <Tab title="failed">
      The transaction could not be completed. Check the `error` object for a stable Capisum error code and message.
    </Tab>
  </Tabs>

  ### Handle the response

  Capisum returns a normalized response body for every terminal status. Parse the `status` field to determine your next action.

  **Settled (success):**

  ```json Settled response
  {
    "id": "txn_8f2k9m",
    "status": "successful",
    "provider": "paystack",
    "provider_ref": "PSK-9281",
    "amount": 500000,
    "currency": "NGN",
    "type": "pay_in",
    "reference": "order-78901",
    "created_at": "2024-01-15T14:02:11Z"
  }
  ```

  When `status` is `"successful"` or `"settled"`, mark the payment as complete in your system and trigger your fulfillment workflow.

  **Failed (insufficient funds):**

  ```json Failed response
  {
    "id": "txn_8f2k9m",
    "status": "failed",
    "provider": "paystack",
    "type": "pay_in",
    "amount": 500000,
    "currency": "NGN",
    "reference": "order-78901",
    "error": {
      "code": "INSUFFICIENT_FUNDS",
      "message": "The customer's account does not have sufficient funds to complete this transaction.",
      "retryable": false
    },
    "created_at": "2024-01-15T14:02:11Z"
  }
  ```

  When `status` is `"failed"`, read `error.code` to determine how to respond. `INSUFFICIENT_FUNDS` is a terminal failure — do not retry it automatically. Surface a clear message to the customer and allow them to try a different payment method. Errors with `"retryable": true` (such as `PROVIDER_TIMEOUT`) can be retried with a new transaction and a fresh reference.

</Steps>

## Error codes reference

| Code | Retryable | Description |
|------|-----------|-------------|
| `INSUFFICIENT_FUNDS` | No | Customer's account balance is too low. |
| `INVALID_ACCOUNT` | No | The account or card details are invalid. |
| `PROVIDER_TIMEOUT` | Yes | The provider did not respond within the allowed window. |
| `PROVIDER_ERROR` | Yes | A transient error on the provider's side. |
| `INVALID_CURRENCY` | No | The currency is not supported by the selected provider. |
