The Capisum sandbox gives you a full-fidelity testing environment where every API endpoint, provider adapter, and response shape behaves exactly as it does in production — with one critical difference: no real money moves. You can create pay-in and payout transactions, simulate success and failure states, inspect normalized responses, and validate your error-handling logic all before connecting a single live provider credential. The sandbox is the right starting point for any new integration and for regression testing before deploying changes.

## How sandbox mode works

Sandbox mode is determined entirely by your API key — not by a separate base URL or a query parameter. When you authenticate with a test key (`cap_sk_test_...`), every request you make is routed through sandbox adapters that mirror their production counterparts. The base URL stays the same:

```
https://api.capisum.com/v1/
```

All providers available in production have sandbox equivalents. You don't need live provider credentials to use the sandbox — test credentials are managed entirely within Capisum.

<Note>
  Sandbox and production environments are fully isolated. Transactions created with a test key never appear in your production transaction logs, and vice versa. To switch environments, swap your API key — nothing else changes in your request code.
</Note>

## Setup steps

<Steps>
  ### Get your test API key from the dashboard

  Log in to the [Capisum dashboard](https://capisum.com/dashboard) and navigate to **Settings → API Keys**. Under the **Test Keys** section, copy your `cap_sk_test_...` key. If no test key exists yet, click **Generate Test Key** to create one.

  Store the test key in your local environment configuration — for example in a `.env` file — and make sure it is never committed to version control:

  ```bash .env
  CAPISUM_API_KEY=cap_sk_test_xxxxxxxxxxxxxxxxxxxxxxxx
  ```

  ### Connect a provider in sandbox mode

  In the dashboard, go to **Providers** and select the provider you want to test with. Each provider tile shows a **Sandbox** toggle. Enable sandbox mode for that provider — you won't need real API credentials. Capisum supplies its own test adapter credentials internally, so no provider account of your own is required to start testing.

  Once enabled, the provider tile will show a **Sandbox Active** badge. You can connect multiple providers in sandbox mode simultaneously.

  ### Run test transactions using special test values

  Use the test amounts and email addresses in the table below to trigger specific simulated outcomes. Capisum's sandbox adapters inspect these values and return the corresponding response state so you can exercise every code path in your application.

  | Test Value | Simulated Result |
  |------------|------------------|
  | `amount: 100` | Successful transaction |
  | `amount: 200` | Failed — `INSUFFICIENT_FUNDS` |
  | `amount: 300` | Provider timeout — `PROVIDER_TIMEOUT` |
  | `email: "fail@example.com"` | Invalid account — `INVALID_ACCOUNT` error |

  Here's a complete sandbox request that simulates a successful pay-in:

  ```bash cURL (sandbox)
  curl -X POST https://api.capisum.com/v1/transactions \
    -H "Authorization: Bearer cap_sk_test_xxxxxxxxxxxxxxxxxxxxxxxx" \
    -H "Content-Type: application/json" \
    -d '{
      "provider": "paystack",
      "type": "pay_in",
      "amount": 100,
      "currency": "NGN",
      "customer": {
        "email": "customer@example.com",
        "name": "Jane Doe"
      },
      "reference": "test-order-001"
    }'
  ```

  And the same request simulating an `INSUFFICIENT_FUNDS` failure:

  ```bash cURL (sandbox — simulate failure)
  curl -X POST https://api.capisum.com/v1/transactions \
    -H "Authorization: Bearer cap_sk_test_xxxxxxxxxxxxxxxxxxxxxxxx" \
    -H "Content-Type: application/json" \
    -d '{
      "provider": "paystack",
      "type": "pay_in",
      "amount": 200,
      "currency": "NGN",
      "customer": {
        "email": "customer@example.com",
        "name": "Jane Doe"
      },
      "reference": "test-order-002"
    }'
  ```

  ### Review responses in Transaction Logs

  After running test transactions, navigate to **Dashboard → Transactions** and make sure the environment switcher in the top bar is set to **Sandbox**. Every test transaction appears here with full request/response details, provider adapter output, latency, and status — identical to what you'd see for production transactions.

  Use this view to confirm:
  - Your application is sending the correct request payload
  - Error codes are being handled and surfaced correctly in your UI
  - Status polling logic reaches the expected terminal state
  - Your `reference` fields are unique and correctly formatted

</Steps>

## Switching to production

When you're satisfied with your sandbox test results, switching to production requires only two changes:

1. Replace your test key (`cap_sk_test_...`) with your live key (`cap_sk_live_...`)
2. Add your real provider credentials under **Providers → [Provider Name] → Edit Credentials**

No changes to your request structure, endpoint URLs, or response-handling code are needed. The request and response shapes are identical across environments.

<Tip>
  Keep your sandbox environment active even after going live. It's the fastest way to reproduce a reported bug, test a new provider before enabling it in production, or validate a code change without risk to real transactions.
</Tip>
